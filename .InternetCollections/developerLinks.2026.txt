Roland: 

My main gripe with PHPUnit is how to test a piece of my extension which also requires the Joomla codebase

Brian:

Presumably the same problem with cypress

No, that is different: the tests in Cypress run on the Joomla instance in which the extension under test is installed. So the extension just uses a Joomla site that has the whole Joomla codebase. You only test the end result. You script a walk through the browser, giving input where needed, and then look what is displayed on the page (in a virtual browser). 

With Unit testing you test the extension in isolation. If you use other classes from Joomla, you'll have to mock them (make a fake double of them, only giving the output you want them to give when used in your extension). You usually do that in the bootstrap.php file. See for instance in /tests/bootstrap.php in last year's GSoC project for migration between CMSs using the web services API. 

It would be good if we prepare examples of such mocks and stubs, that can be easily reused by other developers. Also avoiding that almost all Joomla classes are mocked and stubbed like in the JA CMS migration project of last year; which is an example of testing if the Joomla classes work as expected; don't do that, only test the functionality that your custom class has added. This year we'll try with some GSoC projects to contribute some more good examples too.
